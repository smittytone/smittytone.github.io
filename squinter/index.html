About Squinter

Introduction

Squinter is a utility to help you work on Electric Imp software development projects using your own editor and code files you keep in a local folder — which may also be sync’d with a cloud service like Dropbox or a version control system like GitHub.

With Squinter and the impCentral API you can:

  • Work on your Electric Imp project code in your favourite text editor
  • ‘Compile’ in local library files before uploading the complete code to the Electric Imp impCloud™
  • You can take charge of your devices:
    • Assign them to application firmware
    • Update their code
    • Reboot devices to run newly uploaded code
    • View device logs
  • And much more...

Incorporating Libraries Into Your Code

Squinter lets you incorporate multiple files into your project. Electric Imp applications — called ‘Models’ — comprise two files: one for device code, the other for cloud-based agent code, but Squinter lets you embed into them links to other files, such as libraries.

For example, you might want to use the same library in multiple models — they all use the same sensor, say. Sharing a single library file allows you to update that file without having to edit every model that uses it. You just recompile each project to incorporate the changes you’ve made.

Squinter expects your model files to be named in the standard Electric Imp schema: *.agent.nut and *.device.nut for, respectively, your agent and device code. The two identifiers (represented here by the wildcard *) need not be identical, but when creating a new project from source code files, Squinter will use the agent code filename as the basis for the project’s name.

Local library files should be entered into your agent and device source code using the following syntax:

#import "library_filepath\library_filename"

or

#include "library_filepath\library_filename"

This is to distinguish these libraries from Electric Imp online libraries, which use the #require directive.

The name of the library file is arbitrary, but *.class.nut and *.library.nut are the recommended forms. You may include a full Unix filepath; if you only provide a filename, Squinter expects the file to reside in the same folder as the project file (see ‘Working with Projects’).

Working With Projects

Squinter works primarily through project files. These record which agent and device code files comprise a given model, and which libraries they use. Project files are identified by the .squirrelproj extension. Squinter lets you create new projects, either from scratch or from existing agent and device code files. You can also create projects from your existing models stored in the Electric Imp impCloud.

Projects are listed in the ‘Currently Open Projects’ submenu in the ‘Project’ menu, which also presents actions you can perform with the currently selected project. You can also select a project from the ‘Current Project’ popup menu in Squinter’s main window.

New projects are saved in Squinter’s working directory, which you can set in the Preferences.

Working With Models

Squinter lists all of the models registered to your Electric Imp account, and you can bind these to existing projects or use them as the basis for new ones. You can delete or rename your models, or by creating a new project, create a new model too. You’ll find these options in the ‘Models’ menu.

Selecting a project will automatically select the model it has been bound to, if it has such a link. You can connect a model to the currently selected project using actions in the ‘Project’ and ‘Models’ menus.

You can assign or re-assign a device to any model at any time using a ‘Models’ menu option.

Working With Devices

Squinter also keeps track of your development devices. It lists these in the ‘Models’ menu’s ‘Current Models’ submenu, alongside the models they are assigned to. Unassigned devices are listed in their own section. Specific devices can also be selected from the ‘Current Device’ popup menu in Squinter’s main window.

In each of these lists, devices that are currently online have a bullet point after their name.

The ‘Device’ menu presents actions you can perform on the currently selected device, including getting information about it, renaming it, removing it from your account and removing its association with a model (unassign it).

You can also force the device to restart. If you want to force all the devices registered to a given model, not just one, there’s an option for that in the ‘Models’ menu.

Debugging Code

You can retrieve the most recent log entries for the device in one go (up to a maximum of 200, all the Electric Imp impCloud retains), or you can stream logs from the device: entries will appear in Squinter’s main window in real time. You can currently only stream from one device at a time.

If the log window becomes too crowded, you can clear it using the ‘Clear’ button in the toolbar, which is also home to a number of shortcuts to various functions provided by Squinter’s menus. You can also print the current contents of the log window if you have a printer.

When you upload code — using the ‘Upload’ button or the ‘Project’ menu option — the Electric Imp impCloud will perform a syntax check. Any errors it spots will be reported in Squinter’s log window. Line numbers refer to the compiled code, so you can use two actions in the ‘View’ menu to list the compiled fully to help you track down the errors.

Configuring Squinter

To take full advantage of Squinter’s features, you will need not only an Electric Imp account but also a Build API key. You can obtain a Build API key by logging in to the Electric Imp IDE via a web browser and selecting ‘Build API Keys’ from the username menu. Paste your API key into the field provided in Squinter’s Preferences panel, accessed through the ‘Squinter’ menu.

The Preferences panel also allows you to select the log window’s font, text size and the colour, and the colour of the background. It provides a number of other useful settings too. You can also choose your main Electric Imp projects folder.
